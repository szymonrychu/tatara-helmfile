- 2026-07-17 (#195/#197/#199/#200) **Removing a CRD field is a TWO-REPO, ORDERED cutover, and `helmfile diff` cannot see either failure.** Removing `spec.scm.cron.mrScan` from the Project CRs cost two failed applies and left the platform paused with both `project-*` releases in `failed` status. Helm 4 builds a TYPED patch object against the live CRD schema; it does NOT rely on apiserver pruning, so it rejects a values/CRD mismatch in BOTH directions. (1) Values removed while the CRD still declares the field: SSA sends `mrScan: null` to delete a field its field manager owns, and the schema says non-nullable object -> `Invalid value: "null": must be of type object`. (2) Values still present after the CRD dropped the field: `failed to create typed patch object: .spec.scm.cron.mrScan: field not declared in schema`. So the values and the CRD MUST move together, operator first: ship the CRD removal, roll it out, THEN drop the values. The reverse order (what we did, on the theory that the operator's pin auto-propagates and would auto-apply the new CRD first) breaks the apply for every release in the bucket, and `--rollback-on-failure` cannot save it because the rollback hits the same typed-patch error. NOTE the asymmetry that misled us: `project-infrastructure` happened to land while `project-tatara` failed, so the first failure looked like a half-applied rollout rather than a schema fight. **`helmfile diff` passed green on BOTH broken states.** Diff does not construct the typed patch, so diff-green does not imply apply-green for a CRD field removal - the #51 diff gate does not cover this class. Recovery: revert the values, apply goes green, roll the operator, then re-land the values removal (#200). Related: tatara-operator MEMORY 2026-07-17 (the claim-lease/handoff-guard change this CRD removal rode along with).

- 2026-06-23 (#51) **PR `helmfile diff` check now BLOCKS the merge on a diff-time error (chart-not-found / render / sops).** The diff job was purely informational: `.github/workflows/diff.yaml` had the diff step `continue-on-error: true`, and `helmfile ... diff --detailed-exitcode ... 2>&1 | tee diff.log` made the shell exit status `tee`'s (always 0), so the helm-diff `--detailed-exitcode` (0 no-change / 2 diff / 1+other error) was discarded - a broken pin (the #45 `0.1.0` placeholder, the #45/#46 GC'd `0.0.0-<sha>` tag) merged green and only died at `apply` on main. Fix is CI-only, three parts: (1) capture `code="${PIPESTATUS[0]}"` as the FIRST command after the `tee` pipeline (any intervening simple command - even `x=$?` - RESETS PIPESTATUS, so only comments may sit between; verified) and expose it via `$GITHUB_OUTPUT`; (2) keep the `sticky-pull-request-comment` step but add `if: always()` so the diff text (incl. the chart-not-found error) still posts when the diff errors, preserving the #47 `path:` fix and the reviewer diagnostic; (3) a separate final `gate merge on diff result` step (`if: always()`) that passes iff `code` is `0` or `2` and fails on `1` / any other / EMPTY value (fail-closed: an unset output blocks). The diff step stays `continue-on-error` so the comment always renders first; the gate step is the sole thing that turns the check red. Out of scope (own follow-ups): `apply.yaml` pre-apply diff/`--wait`/atomic, and shift-left `helm lint`/`template` chart-validate in component repos. LESSON: a pipe to `tee` hides the upstream exit code in `PIPESTATUS[0]`, and `PIPESTATUS` only survives until the next executed command.
